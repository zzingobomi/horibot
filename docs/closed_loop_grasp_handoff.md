# Closed-Loop 파지 구현 사고방식 (다음 구현 모델용)

> 선행 문서(파지 디버깅 post-mortem)를 이미 읽었다는 전제. 거기 있는 사실·기각 결론·실수 목록은
> 반복하지 않는다. 이 문서는 **closed-loop를 어떻게, 어떤 사고방식으로 구현할 것인가**만 다룬다.
> open-loop로는 안 되는 이유는 이미 결론 났다. 여기서부터는 "그래서 loop를 어떻게 짤 것인가"다.
>
> 결정 사항(control law, gain, 끊는 시점 등)은 이 문서의 사고 가이드로 **네가 트레이드오프를
> 계산해서 직접 정하면 된다.** 최종 구현물은 실물 로봇으로 테스트된다.

---

## 0. 핵심 태도

closed-loop는 "코드 한 뭉치"가 아니라 **물체 근처에서 물리 로봇이 도는 제어 루프**다.
루프의 매 iteration마다 스스로에게 물어라:

> **"이 iteration은 지금 무엇이 참이라고 가정하고 있나? 그게 거짓이면 어떻게 되나?"**

이 질문을 기능마다 던지지 않고 코드를 짜면 직전 세션의 참사가 반복된다. 아래 절들은 이 질문을
closed-loop의 각 부분에 적용하는 방법이다.

**기존 시나리오에 갇히지 마라.** pick_and_place 태스크 모듈이 이미 open-loop 시나리오로 구현돼
있지만, closed-loop는 검출→계획→실행이 한 번 흐르는 open-loop와 제어 구조 자체가 다르다(물체
근처에서 측정·명령이 계속 도는 루프). 기존 시나리오의 단계 구성에 억지로 끼워맞추려 하지 마라.
필요하다면 **시나리오 흐름 전체를 다시 설계해도 된다.** 재사용할 것과 새로 짤 것을 구조 관점에서
판단해라 — "기존 코드가 이러니까"가 설계를 제약하게 두지 마라.

---

## 1. 코드 짜기 전에 루프를 **말로** 완전히 정의하라

한 줄도 구현하기 전에 아래를 전부 확정해라. 하나라도 "아마 이럴 것"이면 멈추고 코드를 열어
확인해라 — 짐작으로 넘어가지 마라.

- **무엇을 measure 하는가**: detector가 매 tick 정확히 무엇을 주는가 — pose 6DOF인가 centroid뿐인가,
  어느 frame인가, confidence가 있나, 실패 시 무엇을 반환하나.
- **무엇을 command 하는가**: measure한 오차를 어떤 명령으로 바꾸는가. 로봇이 **주기적 target
  증분(servo-style)** 을 받는 경로가 실제로 있는가? (없으면 그게 first blocker다. 있다면 어느
  frame·rate로 받는지 확인.)
- **loop rate**: 검출(PC) ↔ 명령(pi_hori1)이 분산이다. 한 tick의 예산(검출 지연 + 네트워크 왕복 +
  실행)이 얼마고, 그게 gain·step을 어떻게 제약하나.
- **좌표계 정합**: measure와 command가 **같은 frame 기준으로 닫히는가** (§3).
- **수렴 조건**: 무엇이 "충분히 정렬됨"인가. 정량 기준 (image error < ε, 목표 depth 도달 등).
- **종료·commit**: 언제 servo를 끊고 close로 넘어가는가 (§4).

이걸 다 적으면 그게 설계다. 이 정의가 없으면 edge case를 논할 수도 없다.

## 2. 루프를 상태로 쪼개고, 각 전이가 깨지는 방식을 전부 나열하라

closed-loop 파지는 대략 이런 상태 흐름이다:

```
탐색(물체 찾기) → coarse 정렬(멀리서) → fine 정렬(가까이) → commit(끊고 최종 접근) → close → 검증
```

**각 상태에서, 그리고 각 전이에서 "무엇이 깨질 수 있나"를 먼저 나열하고 각 복구 동작을 정한 뒤에
그 상태의 코드를 짜라.** 이게 이 프로젝트가 요구하는 사고방식이다. 예:

| 상황 | 무엇을 가정했다 깨지나 | 감지 | 복구 (직접 확정) |
| --- | --- | --- | --- |
| 물체가 FOV 부분 이탈(잘림) | "물체는 프레임 안에 온전히 있다" | centroid가 경계 근처 | 잘린 채 진행 vs 프레임 안으로 되당기는 방향 우선 |
| 물체 FOV 완전 이탈 (직전엔 보임) | " " | detection empty + 직전 valid | 마지막 관측으로 open-loop 추정 이동 후 재탐색 |
| 처음부터 못 봄 | "시작 시 물체가 보인다" | 시작 시 empty | 파지 중단, 상위로 반환 |
| 단발 프레임 드롭 vs 진짜 소실 | "empty = 물체 없음" | N-tick 연속 여부 | 단발은 직전값 hold, 연속이면 위로 |
| 가까이서 depth 죽음 (근접 한계) | "가까울수록 측정이 좋다" | depth valid% 급락 / 거리 < 한계 | 한계 전에 servo 끊고 blind 최종 접근 |
| 수렴 안 함 / 발진 | "오차는 단조 감소한다" | error 정체/진동 | gain 감소, iteration·시간 상한 후 abort |
| servo 궤적이 joint limit/singularity | "명령한 증분은 항상 실행된다" | IK 실패/근접 | 경로 재계획 or abort (도달 불가로 속단 금지) |
| 그리퍼 몸통이 바닥·주변 침범 | "tcp만 안 부딪히면 된다" | 근접(margin)+몸통 충돌 체크 | tilt·접근 방향 변경 |
| close 후 EMPTY | "정렬됐으니 잡혔다" | 그리퍼 load 신호 | 재시도 상한, 재검출 후 retry or 실패 보고 |
| 전체 timeout | "언젠가 수렴한다" | 누적 시간 상한 | 안전 정지 + 실패 보고 |

> 위는 시작점이다. **빠진 케이스를 추가하고 각 복구 동작을 확정**하는 것이 설계의 실체다.
> 어떤 상태든 "감지 못 하면 그냥 크래시/무한대기"로 두지 마라 — 모든 실패에 정의된 동작이 있어야
> 한다. 특히 **empty를 어떻게 해석하느냐**(드롭 vs 소실)가 루프 안정성을 좌우한다.

## 3. 좌표계: common-mode 상쇄가 성립하도록 닫아라

closed-loop가 이 로봇에서 먹히는 이유는 오차가 거리비례(각도 성격)라, **물체 가까이서 loop를
닫으면 오차가 줄기** 때문이다. 그리고 eye-in-hand의 진짜 이점은 — **측정과 실행이 같은 자세의
FK 오차를 공유해서 common-mode로 상쇄**되는 것이다.

- 이 상쇄가 성립하려면 **측정한 그 자세를 기준으로 상대 증분을 명령**해야 한다. 검출 자세와
  명령 기준이 어긋나면(직전 파이프라인이 그랬다: 검출 자세 ≠ 파지 자세) 상쇄가 깨지고 절대
  정확도 오차가 그대로 실행에 들어간다.
- 모든 pose에 frame 접미사를 붙여 써라 (`T_base_tcp`, `T_tcp_cam`, `T_cam_obj`). "물체 위치"처럼
  frame 없는 표현 금지. 오차·측정·명령이 각각 어느 frame인지 헷갈리는 순간 버그가 들어온다.
- hand_eye는 부정확하다(자세의존 잔차 남음). **closed-loop의 목적은 이 부정확을 feedback으로
  흡수하는 것**이지, hand_eye를 정답으로 믿는 게 아니다. 그러니 절대 pose(base frame)로 목표를
  잡기보다, 카메라가 지금 보는 상대 오차를 줄이는 쪽으로 설계해라.

---

## 4. 마지막 구간과 commit — 여기서 성패가 갈린다

- 가까이 갈수록 오차는 주는데, 어느 순간 물체가 카메라 **근접 한계** 안으로 들어와 depth가 죽거나
  FOV를 벗어난다. 그 전에 **servo를 끊고 최종 궤적을 고정**해야 한다(blind final approach).
- **어느 자세에서 끊느냐가 최종 정확도를 결정한다.** 이 지점의 관측이 마지막 정보다. 너무 일찍
  끊으면 절대오차가 남고, 너무 늦게 끊으면 depth 소실 상태에서 잘못된 증분을 명령한다. 이 트레이드
  오프를 명시적으로 결정해라.
- close 직전/직후 그리퍼 load로 검증. EMPTY면 재시도 정책에 따라 재검출부터.

## 5. control law 자체는 §1~§4를 채운 뒤에 고른다

IBVS(image feature 오차로 제어)냐 PBVS(추정 pose 오차로 제어)냐, gain·step은 얼마냐는 — 위의
측정 계약·loop rate·좌표계·근접 한계가 정해지고 나서야 근거 있게 고를 수 있다. **먼저 고르고
끼워맞추지 마라.** 이 로봇은 pose 추정이 자세의존적으로 틀리므로, 추정 pose를 덜 믿는 방식
(image 오차 직접 이용)이 초기 검토 가치가 있다.

> 착수 전 권장: 정지 큐브를 여러 자세에서 찍은 offline 데이터로 **cam-frame centroid가 자세가
> 바뀌어도 안정적인지** 먼저 확인. centroid가 튀면 어떤 control law도 그 위에서 못 선다.

## 6. 성공을 목표로 짜되, 실패하면 **분석 가능하게** 남겨라

처음부터 집는 걸 목표로 설계해라 — "어차피 초반엔 실패하겠지"라는 전제로 대충 짜지 마라.
그러면서도, 만약 실패했을 때 **"데이터가 없어서 왜 실패했는지 분석이 안 됩니다"가 나오면 안
된다.** 사후분석에 필요한 걸 처음부터 전부 남기도록 설계에 로깅을 넣어라. 이건 선택이 아니라
필수 요구사항이다.

- **매 tick 트레이스**: timestamp, detection 원출력(pose/centroid/confidence/valid%), 계산된 오차,
  명령한 증분, 그때의 joint 각·`T_base_tcp`, 사용한 frame들. loop가 왜 그 방향으로 움직였는지를
  나중에 tick 단위로 재구성할 수 있어야 한다.
- **결정 지점 기록**: 어느 상태였는지, 어떤 edge case 분기를 탔는지(§2 표의 어느 행), servo를
  **어느 자세에서 끊었는지**(commit 지점), close 시 load 신호, 최종 성공/EMPTY.
- **원시 관측 저장**: 실패 분석에 필요한 raw depth / mask / detection json / 필요시 point cloud를
  기존 debug 폴더 관례(`debug/detect/<timestamp>/` 에 json+raw depth+mask+ply)와 **같은 구조로**
  남겨라. 특히 commit 직전 프레임과 실패 직전 몇 tick은 반드시.
- **run 단위 요약**: 시도마다 한 폴더에 위를 묶고, 결과(성공/실패 + 실패 시 어느 단계에서)를
  한눈에 볼 수 있는 요약을 남겨라. 그래야 여러 시도를 나중에 비교·분석할 수 있다.

기준: **로그와 debug 폴더만 보고도 "이 시도가 왜 실패했는지"를 하드웨어 없이 재구성할 수 있어야
한다.** 그게 안 되면 로깅이 부족한 것이다.

---

## 7. 완료 정의

**실제 2.5cm 큐브를 반복해서 집어 올리는 것**이 완료다. 성공률로 보고하면 좋다(예: 10회 중 N회),
어떤 조건에서 되고 안 되는지도 함께. 중간 진전(정렬 품질 개선, 실패 단계가 뒤로 밀림 등)은 정직하게
공유해라 — 무엇이 나아지고 무엇이 남았는지 보이는 게 도움이 된다. 다만 "루프 완성"과 "파지 성공"은
다른 것이니 구분해서 말하면 된다.
