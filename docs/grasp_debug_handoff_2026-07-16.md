## 1. 목표와 현재 상태

- **목표**: SO-101 (6DOF, D405 wrist eye-in-hand) 로 책상 위 **2.5cm 흰 큐브(주사위 무늬 있음)** 를
  pick_and_place 로 집기.
- **현재 상태**: **성공 0번.** 이 세션 시작 시점에도, 끝날 때도 못 집음.
- 시스템: 분산. **motion+motor = pi_hori1**, **detector/pick_and_place/scene3d/calibration/bridge = PC**.
  DB(`backend/horibot.db`)는 git tracked, 각 머신 로컬 복사.

### 파지 실패의 마지막 관측된 형태 (세션 끝 기준)
- 인식은 단일 뷰 기준 대략 맞음(큐브 바닥 z≈0, height≈2.5cm).
- 그러나 **멀티뷰 fusion 이 49×56×41mm 덩어리**(큐브는 25mm)로 뭉개짐 → 파지 기하 오염.
- 파지 실행: tilt+45(위에서 비스듬히)로 감, **팔 도달 오차 12.8mm**, close 후 **EMPTY**(물체 없음).

## 2. 검증된 사실 (데이터로 확인 — 믿어도 됨)

1. **텍스처/센서 near-bias 가설은 틀렸다.** 큐브 mask 안 raw depth 는 **valid 96~100%, std 5~7mm**
   로 조밀·저노이즈. 저텍스처 dropout 의 신호가 아니다. (옛 grasping.md §2 의 "D405 저텍스처
   depth 편향" 결론은 **오답**.)
2. **재구성(FK∘hand_eye∘depth)이 계통적으로 기울고(3~7°) 떠 있었다(+15~20mm).** 큐브 옆 **테이블
   자체가** 100mm 밖까지 같은 양으로 떠 있음(object-specific 아님). → 캘 외부변환 오차.
3. **rigid transform 은 크기 불변** → base frame 에서 큐브 extent 가 뷰마다 다르면, 그건 **카메라
   frame 점군 자체가 다른 것**(캘이 늘린 게 아님). 확인: cam-frame 큐브 깊이 extent 가 뷰마다
   20mm(0012)~56mm(0004)로 제각각.
4. **정지한 같은 큐브가 카메라 자세마다 다른 base 위치/크기로 찍힌다.** 중심 ±7~10mm, extent
   20~56mm. **거리에 비례**(카메라 30cm → 오차 ~40mm, 15cm → ~1cm) = 잔여 **회전 오차의 lever arm**.
5. **FK 체인은 내부적으로 일관.** offline BA(hand_eye 산출) FK 와 v2 motion FK 를 7/6 관절각으로
   대조 → **0.05° 일치**. FkChain(analytic) vs PyBullet 엔진도 **0.01° 일치**. URDF `tcp` 프레임도
   6/21↔현재 동일. → **"v2 포팅 FK 회귀"는 아니다.**
6. **hand_eye 값은 6/21 이후 안 바뀌었다**(offline_BA_stage_D). 7/6 엔 라이브 클라우드가 평평했다고
   사용자가 기억. (이 모순은 §6-C 참조 — 완전 규명 못 함.)
7. **SOR(statistical outlier removal)은 덩어리를 못 고친다.** 검증: 0004 를 SOR 해도 40mm→40mm.
   퍼짐이 sparse outlier 가 아니라 **조밀**하기 때문.
8. **팔 도달(reach) 오차 ~1cm** (마지막 12.8mm, 이전 6.8/3.7mm). open-loop 백래시/FK.
9. **종합 오차 ≈ 1~2cm ≈ 큐브 크기(2.5cm).** open-loop 파지가 구조적으로 아슬아슬.
10. **그리퍼 tcp 는 손끝에 맞다**(mesh 로 tcp 기준 x≈+1mm). 단 **몸통이 tcp 뒤로 91mm, z±24mm**
    두껍다 → tilt 큰(near-horizontal) 파지 시 몸통이 바닥 침범.
11. **바닥충돌 체크는 관통(c[8]<0)만 봤다 → 몸통이 바닥 스치는 걸 놓침.** (이 세션에서 margin
    6mm + 그리퍼 몸통 반영으로 수정 — §7.)

## 3. 근본 원인 (현재 최선의 이해)

**로봇의 기구학 정확도(FK+hand_eye, 자세의존)가 ~1cm, 팔 도달이 ~1.3cm 이고, 큐브가 2.5cm.
즉 오차가 물체 크기와 맞먹는다.** 이게 두 방식으로 파지를 깬다:

- **멀티뷰 정렬 실패**: 각 뷰 점군은 "그때 카메라가 어디였나(FK+hand_eye)"로 base 에 놓이는데,
  그게 뷰마다 ~1cm 틀려서 **같은 물체를 본 뷰들이 서로 어긋나 놓임** → 겹치면 또렷해지는 게
  아니라 4~5cm 덩어리로 퍼짐.
- **open-loop 파지 목표 오차**: 파지점이 ~1~2cm 빗나감.

**캘은 정밀(σ_R 0.8°/σ_t 7.5mm)하지만 자세의존적으로 부정확하다.** σ(정밀도)를 아무리 짜내도
정확도는 안 잡힌다. **고정 보정(fixed hand_eye patch)으로는 자세의존 오차를 못 없앤다.**

> ⚠️ 이건 "확정된 단일 근본"이 아니라 **여러 증거가 수렴하는 최선의 이해**다. 잔여 오차가
> hand_eye 회전이냐 FK(관절/백래시)냐의 정확한 분해는 **못 끝냈다**(touch test 가 노이즈 지배).

## 4. 내가(Claude) 반복한 실수 — 다음 모델은 하지 마라

1. **Reactive(수동적).** 코드를 확인 안 하고 짐작으로 단언 → 사용자가 짚으면 그제서야 열어봄.
   예: "멀리서 본 뷰를 파지에 쓴다"고 단정 → 확인해보니 이미 close 뷰만 씀(내가 틀림).
   **교훈: 시스템 동작에 대한 주장은 반드시 코드/데이터를 먼저 열어 확인한 것만.**
2. **과신한 오답 연발**: "텍스처가 문제"(오답) → "v2 회귀다"(오답) → "floor margin 이면 될 것"
   (파지 실패) → "closed-loop 이 진짜 답"(미검증인데 단정) → "단일 뷰 쓰자"(품질 판단 불가).
   **교훈: 검증 전엔 "이거 하면 된다" 금지.**
3. **성공 0번인데 "진전 있다"고 치어리딩.** 사용자를 더 화나게 함.
   **교훈: 실제 목표(파지) 성공 전엔 진전 운운 금지. 정직하게 "아직 0번".**
4. **숨은 가정 주입**: "물체는 책상 위"(support-plane), "큐브는 25mm 콤팩트"(뷰 outlier 판단),
   "IK 파지 불가 = 물리적 도달 불가"(사용자가 오프라인으로 도달 가능 실증). §6 참조.
5. **검증 없이 fix 제안.** (floor margin 은 검증했으나 결과 파지 실패, SOR 은 검증해서 기각 —
   후반엔 개선됨. 전반엔 안 함.)
6. **혼란스러운/부정확한 워딩**: "파지점이 큐브 정중앙에 간다"(실제론 조가 옆면에 감) 같은 표현으로
   사용자 헷갈리게 함.
7. **잘못된 배포 안내**: floor_collision 은 pi_hori1(motion)에서 도는데 "PC 재시작"이라고 함.

**메타 실수**: 증상마다 즉흥 스크립트를 쏟아내며 국소적으로 파느라, "이 로봇으로 물체를
어떻게 집어야 하나"라는 **큰 그림/상식**을 자꾸 놓침. 사용자가 계속 "상식적으로 생각하라"고 함.

## 5. 시도한 접근과 실패 이유

| 시도 | 결과 | 왜 실패/무효 |
| --- | --- | --- |
| 텍스처 A/B (큐브에 점) | 실패 | depth 는 원래 깨끗함(§2-1) — 텍스처가 원인이 아니었음 |
| "v2 FK 회귀" 추적 | 기각 | FK 체인 내부 일관(§2-5), hand_eye/URDF 불변 |
| **hand_eye 보정 유도+적용** (δR7.77°, δt(11,3,12)mm) | **부분 성공** | 테이블 평평해짐(하드웨어 확인) — 평균 기울기/부양은 잡음. **그러나 자세의존 잔차(~1cm)는 못 잡음** → 파지 여전히 실패 |
| floor_collision margin (6mm+몸통) | 배포 전 | 몸통 바닥충돌 파지 기각 → tilt+75(바닥침범) 대신 tilt+45 선택됨. 하지만 파지는 여전히 EMPTY(기하 오염) |
| SOR outlier 제거 | 검증 후 기각 | 퍼짐이 sparse 아니라 조밀 — SOR 로 안 줄음(§2-7) |
| "단일 좋은 뷰만 쓰기" | 기각 | (a) 어느 뷰가 깨끗한지 런타임 판단 불가, (b) 물체 진짜 크기/모양 모름 → 크기로 outlier 판단 불가, (c) 단일 뷰는 반대편 면을 못 봐 antipodal 불가 |
| consensus/뷰-간 정렬(ICP류) | 미착수 | 방향만 제시. 검증 안 함 |

**공통 실패 축**: 위 대부분이 **"정적 인식을 더 똑똑하게 후처리"** 로 ~1cm 자세의존 오차를
없애려 한 것. 근본(기구학 정확도)을 안 건드리면 후처리로는 못 넘는다.

## 6. 틀렸던 가정들 (명시)

- **A. "저텍스처 depth near-bias"** (옛 문서에서 상속) → **틀림.** depth 깨끗, 오차는 캘 외부변환.
- **B. "캘 σ 좋음 = 캘 정확함"** → **틀림.** 정밀도 ≠ 정확도. 자기일관(σ 작음)이면서 절대적으로
  자세의존 틀어질 수 있음. σ 를 더 짜내도 무의미.
- **C. "고정 hand_eye 보정이면 인식 고쳐짐"** → **부분만.** 평균은 잡았으나 **자세의존 잔차 못 잡음.**
  (그리고 7/6 평평했다는 기억 vs 6/21 hand_eye 불변의 모순은 **미규명** — 가능성: 7/6 은 scan
  ICP 정합본이라 per-view 오차가 가려졌거나, 기억 부정확. 확정 못 함.)
- **D. "IK 파지 불가 = 물리적 도달 불가"** → **틀림.** 사용자가 오프라인(토크오프)로 그 위치·자세
  도달 가능 실증. planner 의 "파지 불가"는 도달 가능한 해를 못 찾거나 나쁜 fallback 을 고른 것.
  (메모리 `feedback_verify_solver_not_reality` 그대로.)
- **E. "물체 크기/모양을 알고 인식 품질을 판정할 수 있다"** → **틀림.** ground truth 없음. 물체가
  진짜 길쭉하면 "40mm 로 보이니 outlier"라는 판단이 맞는 뷰를 버리는 것. **모양 가정 금지.**
- **F. "단일 뷰로 충분/판정 가능"** → **틀림.** 한 뷰는 반대편 면 미관측(antipodal 불가) + 품질
  자기판정 불가. **멀티뷰는 목적이 옳다**(반대편 면 관측). 문제는 뷰-간 **정렬**.

## 8. 다음 세션이 할 것 / 하지 말 것

### 하지 말 것 (반복 금지)
- 텍스처/센서 near-bias 재론(§6-A 기각).
- σ 재짜기(§6-B).
- 정적 인식 후처리로 ~1cm 자세의존 오차 없애려 하기(§5 공통 실패축).
- 물체 모양/크기 가정한 outlier 판정(§6-E).
- 단일 뷰 파지(§6-F).
- "IK 파지 불가"를 물리 한계로 받아들이기(§6-D — 도달 가능함, 실증됨).
- 코드 확인 전 단정(§4-1).

### 실제로 파고들 값진 방향 (전부 미검증 — 검증부터)
1. **자세의존 오차의 정확한 분해**: 정지 큐브를 여러 통제된 자세에서 관측하고, per-view 오차를
   관절/자세와 대조 → hand_eye 회전 잔차 vs FK(관절/백래시)로 분리. (touch test 는 손 노이즈 지배로
   실패했음 — 통제된 재관측 필요.)
2. **뷰-간 정렬을 FK 안 믿고 형상으로**(ICP류 / point-level 합의). 모양 무가정. 단 부분뷰 overlap
   작아 어려울 수 있음 — **오프라인 실데이터로 먼저 검증.**
3. **피드백 기반** (사용자 반응은 회의적이었으나 논리적 귀결):
   - **집는 자세에서 검출**(eye-in-hand 공통오차 상쇄) — 검출 오차와 실행 오차가 같은 관절자세라
     상쇄. 지금 파이프라인은 검출 자세 ≠ 파지 자세라 상쇄 안 됨.
   - closed-loop visual servoing / 촉각(그리퍼 load — 이미 verify_grasp 로 있음).
4. **정확도-지향 재캘** (σ 아니라 절대정확도 기준. 예: 알려진 평면/기준점 구속). 단 사용자는 재캘에
   지쳐 있음 — 신중히.

### 핵심 판단 (다음 모델이 붙들 것)
- 정답(물체 ground truth)을 모르는 상태에서 **정적 측정 하나로 "맞다"를 보증할 방법은 없다.**
  가정 없이 강건하려면 **cross-view 일치** 또는 **피드백**뿐. (사용자와 함께 도달한 결론.)
- open-loop 정확도 바닥 ≈ 큐브 크기(메모리 `project_grasp_openloop_accuracy_floor` 와 일치).

## 9. 검증 자산 (`backend/scripts/grasp_verify/`)
